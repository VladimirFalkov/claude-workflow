# Audit cycle — формат ручного аудита и фиксов

Этот skill описывает **ручной аудит Владимира** кода от Claude. Это отличается от subagent `reviewer` — который делает автоматическое ревью по `git diff` и применяется по триггеру «дальше».

## Два типа ревью

| | `reviewer` subagent | Ручной аудит Владимира |
|---|---|---|
| Кто запускает | Claude по триггеру «дальше» | Владимир пишет замечания в чат |
| Когда | После принятия задачи, перед коммитом | Сразу после выдачи кода от Claude |
| Формат | [BLOCKER]/[MAJOR]/[MINOR]/[NIT] в 05-review.md | 🔴🟡🟢 в тексте сообщения |
| Цель | Независимая проверка по спеке | Быстрая итерация над свежим кодом |

**Оба типа не взаимоисключают.** Владимир обычно делает ручной аудит после каждой выдачи кода, а `reviewer` прогоняется один раз в конце задачи (по «дальше»).

## Формат ручного аудита Владимира

Владимир присылает замечания **с маркерами приоритета**:

### 🔴 Критическое — исправляем ОБЯЗАТЕЛЬНО
Security-дыры, баги, сломанная логика, серьёзные архитектурные проблемы.

### 🟡 Важное — исправляем ЕСЛИ БЫСТРО
Code smells, subtle bugs, missing error handling. Фиксим, если займёт ≤ 5 минут. Иначе — в backlog.

### 🟢 Косметика — НЕ ТРАТИМ ВРЕМЯ
Если Владимир пометил 🟢, он не хочет тратить токены. **Claude не показывает, не фиксит, не комментирует.**

### Формат замечания

Каждое замечание содержит:
1. **Описание проблемы** — что не так, где (файл:строка)
2. **Варианты решения А/Б** — опционально, если есть развилка
3. **Рекомендация Владимира** — какой вариант он видит лучше и почему

Пример замечания от Владимира:

```
🔴 src/measurements/services.py:42 — коммит в сервисе. По конвенции 
проекта commit принадлежит роутеру, сервис только flush.
А) убрать commit, оставить только flush
Б) вынести всю логику в роутер
Рекомендую А — структура сервиса норм, только убрать лишнюю строку.

🟡 src/measurements/schemas.py:15 — нет extra="forbid" на Create-схеме.
А) добавить ConfigDict(extra="forbid")
Б) оставить как есть — это internal API
Рекомендую А, у нас extra="forbid" на всех публичных схемах.
```

## Поведение Claude во время аудита

### Правило №1: МОЛЧАТЬ пока аудит идёт

Когда Владимир начал писать замечания — он может продолжать в следующих сообщениях. **Claude не начинает писать фиксы на первое же замечание.**

Ждёт явного сигнала:
- Фразы «вот и всё», «это всё замечания», «применяй»
- Или нескольких сообщений подряд от Владимира без новых замечаний (значит закончил)
- Или явной команды «фикси», «применяй фиксы»

Если непонятно — **спросить коротко**: «Ещё замечания, или применять?»

### Правило №2: Код во время аудита НЕ ПИШЕТСЯ

Во время фазы аудита Claude **только слушает**. Не защищается, не объясняется, не пишет код.

Если Владимир явно просит уточнение — отвечает коротко, без кода.

### Правило №3: Игнорировать 🟢

Если в замечаниях есть 🟢 пункт — Claude:
- **Не применяет** его
- **Не комментирует** его
- **Не упоминает** в ответе
- Просто фильтрует список до 🔴 и 🟡

## Применение фиксов

После явного «применяй»/«фикси»/конца аудита:

### Шаг 1 — Обработать по приоритету

1. Сначала 🔴 (все подряд, без обсуждений — они обязательны)
2. Потом 🟡 (если Владимир не сказал «только красные»)
3. 🟢 — НЕ ТРОГАТЬ

### Шаг 2 — Формат фикса

Используй формат из skill `sprint-workflow`:

```
Файл: src/measurements/services.py
Строка: ~42 (функция create_reading)

Было:
    await db.commit()
    return reading

Стало:
    await db.flush()
    return reading
```

**Важно:** первая строка всегда `Файл: ...`, даже если очевидно из контекста.

### Шаг 3 — Targeted fixes, не переписывание файла

По умолчанию давай **только изменённые куски**. Полный текст файла — **только если правок очень много** (более 5-7 в одном файле или они разбросаны по всему файлу).

### Шаг 4 — Один файл — один блок кода

Если фиксы в 3 файлах — 3 отдельных блока. Никогда не объединяй.

### Шаг 5 — После всех фиксов — короткое резюме

В конце сообщения с фиксами — короткое summary:

```
Применено: 2 красных, 1 жёлтый.
Не применено: 1 жёлтый (откладываем в backlog по твоей рекомендации).
```

Без длинных объяснений — Владимир сам увидит, что сделано.

## Несогласие с замечанием

Если Claude считает, что замечание **ошибочно** или **рекомендация Владимира не оптимальна** — говорит **до** применения:

```
🔴 замечание про commit в сервисе — согласен, применяю А.

🟡 замечание про extra="forbid" — уточню перед применением: 
схема используется и публично (API POST), и внутренне (импорт из XLSX).
Если применю extra="forbid", сломается импорт — там клиент передаёт 
несколько служебных полей. Варианты:
  - Добавить extra="forbid" только на Create-схему роута, для импорта 
    сделать отдельную схему
  - Оставить extra="allow" и добавить явные поля
Что выбираем?
```

**Claude не делает молча то, что считает неправильным.** Но и не спорит долго — одно уточнение, решение Владимира, применение.

---

## Тест-рекомендации — НЕ В АУДИТ

В аудит Claude **не предлагает** писать тесты. Это отдельный этап.

Если Claude видит, что нужно добавить тест — говорит **отдельно**, после аудита:

```
Замечания по коду применил.

Отдельно: для сценария "timeout при charge" сейчас нет теста. 
Добавить сейчас или в backlog?
```

---

## Багфиксы во время тестирования (после «дальше»)

Если Владимир нашёл баг при ручном тестировании **после** того, как сказал «дальше» — это **отдельный цикл аудита**:

1. Баг описывается в обычном формате (🔴/🟡)
2. Claude применяет фикс как обычно
3. **Но**: этот фикс идёт **отдельным коммитом** после основного (не мёржится в предыдущий)
4. В `DONE.md` при следующем «дальше» — упоминается как «исправлен баг, найденный при тестировании»

## Anti-patterns — НЕ ДЕЛАЙ ТАК

❌ Начать писать фиксы сразу на первое замечание, пока Владимир ещё пишет остальные
❌ Показать 🟢 замечание или обсудить его
❌ Объяснять, почему Claude изначально написал «неправильно» — это пустые токены
❌ Защищаться от замечания без конкретного технического аргумента
❌ Переписать файл целиком, если достаточно 2 точечных изменений
❌ Объединить фиксы нескольких файлов в один блок кода
❌ Применять 🟡 замечание, если Владимир сказал «только красные»
