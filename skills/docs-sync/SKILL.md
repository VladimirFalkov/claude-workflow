# Docs sync — синхронизация документации с кодом

Этот skill срабатывает **перед коммитом**. Его задача — гарантировать, что пользовательская и dev-документация не расходится с кодом. Работает в связке с `sprint-workflow` (триггер «коммить») и `git-commits` (финальный коммит).

## Основная философия

Документация должна быть **синхронна с кодом в каждом коммите**. Если diff меняет документированное поведение — правка docs уезжает в тот же коммит. Никогда не «обновлю потом» — это путь к устаревшей документации через три месяца.

## Два типа документации

### docs/user/Manual.md — АКТУАЛЬНОЕ состояние продукта

- Для **пользователя** (не разработчика)
- Отвечает на вопрос: «как этим пользоваться прямо сейчас?»
- Обновляется **in-place** — старые формулировки **переписываются**, не дополняются
- Пример: «Мини-карта включена по умолчанию» → после фичи скрытия → «Мини-карта по умолчанию скрыта, включается в меню вида»

### docs/dev/*.md — АКТУАЛЬНАЯ разработческая документация

- Для **разработчика** (твой будущий я через 6 месяцев)
- `architecture.md`, `data-model.md`, `patterns.md`
- Обновляется **in-place** при архитектурных изменениях
- Пример: добавили новую таблицу — в `data-model.md` появляется описание таблицы и связей

### Что НЕ относится к docs-sync

- `.project-state/sprints/spN/RELEASE.md` — его ведёт `task-tracking`, не `docs-sync`
- `docs/user/changelog.md` — его обновляет `task-tracking` при «закрываем спринт»
- `docs/adr/NNNN-*.md` — неизменные, только создаются новые
- `README.md` — обновляется вручную Владимиром при релизах

## Workflow при триггере «коммить»

### Шаг 1 — Понять, что в diff

Запросить `git diff --staged` (если нет staged — `git diff`). Проанализировать:

- **Какие файлы изменены?** — определяет тип изменения
- **Что именно изменилось?** — функциональность, поведение, UI, схема БД, API?

### Шаг 2 — Матрица релевантности

| Тип изменения в коде | Релевантен Manual.md? | Релевантен dev docs? |
|---|---|---|
| Новая view/page/component в UI | ✅ почти всегда | ⚠️ если добавляет архитектурный паттерн |
| Новый API endpoint | ⚠️ если публичный | ✅ в architecture.md |
| Изменение модели (поле, enum, связь) | ⚠️ если это видно в UI | ✅ в data-model.md |
| Новая миграция | ❌ нет | ✅ если меняется схема |
| Изменение дефолтного значения настройки | ✅ **всегда** | ⚠️ если архитектурное |
| Изменение permissions/roles | ✅ если пользователь заметит | ✅ в patterns.md или dedicated |
| Рефакторинг без изменения поведения | ❌ нет | ⚠️ если меняет паттерн/конвенцию |
| Багфикс (поведение приводит к документированному) | ⚠️ если документация описывала **сломанное** поведение | ❌ обычно нет |
| Багфикс (поведение уходит от документированного) | ✅ **всегда** | ❌ обычно нет |
| Новая зависимость, CI-изменения | ❌ нет | ⚠️ если меняет setup |
| Форматирование, типы, docstrings | ❌ нет | ❌ нет |

### Шаг 3 — Решить, проверять ли docs

Если по матрице **всё помечено ❌** — docs-sync **молча пропускает**, передаёт управление обратно в `git-commits`.

Если хотя бы один ✅ или ⚠️ — продолжаем.

### Шаг 4 — Прочитать docs, если они существуют

Для каждого релевантного файла:
- `docs/user/Manual.md` — если затронут пользовательский UX
- `docs/dev/architecture.md` — если меняется архитектурный паттерн
- `docs/dev/data-model.md` — если меняется модель данных
- `docs/dev/patterns.md` — если меняется проектная конвенция

**Если файла нет** — не создавать автоматически. Спросить: «Manual.md пока не создан. Создать и добавить туда описание новой фичи?»

### Шаг 5 — Найти упоминания затронутой функциональности

Для каждого релевантного файла — Grep по ключевым словам из diff:
- Имена новых/изменённых полей, функций, классов
- Названия UI-элементов, пунктов меню
- Термины из бизнес-логики

### Шаг 6 — Показать Владимиру предложение (обязательно ВСЕГДА)

Формат показа:

```
Нашёл в docs/user/Manual.md упоминание "Мини-карта":

Текущий текст (строки 42-44):
> Мини-карта показывается по умолчанию в правом нижнем углу.
> Её можно скрыть через настройки вида.

После твоего изменения поведение стало другим (мини-карта по умолчанию скрыта).

Предлагаю заменить на:
> Мини-карта по умолчанию скрыта.
> Её можно включить через меню вида → "Показать мини-карту".

Варианты:
  1. Применить как предложено
  2. Правлю иначе: [свой текст]
  3. Пропустить (оставлю текущий текст)
  4. Показать полный контекст файла

Выбираешь?
```

**Никаких автоматических правок.** Только после явного «1», «применить», «да».

### Шаг 7 — Применить по порядку

Если найдено несколько мест, требующих обновления — **идти по одному**, не скопом. Каждое — отдельное подтверждение.

Это важно: Владимир может одно принять, другое скорректировать, третье пропустить.

### Шаг 8 — Поставить в staging и передать в git-commits

После всех правок:
```bash
git add <обновлённые docs>
```

В ответе Владимиру:

```
Обновлено в docs:
- docs/user/Manual.md (раздел "Мини-карта")
- docs/dev/architecture.md (секция "Settings defaults")

Всё в staging. Передаю в git-commits для формирования commit message.
```

После этого управление возвращается в `git-commits` skill, который формирует сообщение коммита **с учётом того, что docs тоже изменены**.

## Особые случаи

### Manual.md не существует

Если `docs/user/Manual.md` нет, но изменение значимое для пользователя — спросить:

```
Manual.md не существует. Изменение про "мини-карту" — значимое для пользователя.

Варианты:
  1. Создать docs/user/Manual.md и добавить раздел "Интерфейс карты → Мини-карта"
  2. Не создавать сейчас (Manual напишем позже единоразово)
  3. Только в RELEASE.md, в Manual не пишем

Выбираешь?
```

### dev docs не существует

Аналогично — спросить, создавать или нет. **Но dev docs обычно проще создавать лениво** (создаём когда реально пишем большой объём), поэтому дефолт-рекомендация — пропустить для мелких правок.

### Изменение задокументировано как BREAKING

Если в коммите есть `BREAKING CHANGE` (по `git-commits` skill) — docs-sync **обязан** поднять тревогу:

```
⚠️ В commit message помечено BREAKING CHANGE про переименование depth_cm → depth_mm.

Manual.md содержит упоминание "сантиметры" в 3 местах:
- Строка 15: "глубина в сантиметрах"
- Строка 34: "от 0 до 100 см"
- Строка 78: "см" в таблице единиц

Все 3 — обновить? Показать по одному?
```

### Конфликт между RELEASE.md и Manual.md

RELEASE.md — история изменения («что сделали в этом спринте»).
Manual.md — актуальное состояние («как это работает сейчас»).

Одна и та же фича в обоих **формулируется по-разному**:

- **RELEASE.md:** «Изменено поведение: мини-карта теперь по умолчанию скрыта»
- **Manual.md:** «Мини-карта по умолчанию скрыта. Включается через меню вида»

docs-sync работает только с Manual.md. За RELEASE.md отвечает task-tracking.

## Anti-patterns

❌ **Молча править Manual.md** — всегда только после явного подтверждения Владимира
❌ **Предлагать обновление там, где код не менял поведения** — это лишние токены, раздражает
❌ **Смешивать Manual.md и RELEASE.md** — это разные файлы с разной логикой
❌ **Дописывать в Manual.md вместо замены** — Manual всегда актуальное состояние, старое заменяется
❌ **Править ADR** (`docs/adr/*`) — они неизменны, только создаём новые с Supersedes
❌ **Создавать Manual.md автоматически без спросу** — спрашивать всегда
❌ **Откладывать обновление docs «до следующего коммита»** — нет, либо в этот коммит, либо в backlog

---

## Когда docs-sync НЕ СРАБАТЫВАЕТ

- Триггер другой (не «коммит»)
- Все файлы в diff попадают в категорию «не релевантно» из матрицы
- Владимир явно говорит «коммить без проверки docs» / «не лезь в docs» / «быстрый коммит»

В этих случаях skill молча пропускает выполнение и не спрашивает лишнего.

## Пример полного цикла

```
Ты: (после завершения задачи "сделать minimap скрытым по умолчанию")
Ты: коммить

Claude (git-commits skill активируется):
  Анализирую git diff --staged...
  Активирую docs-sync для проверки документации.

Claude (docs-sync skill активируется):
  В diff вижу изменение:
    src/settings/defaults.py: minimap_enabled=True → False

  Проверяю docs/user/Manual.md...
  Нашёл 1 упоминание "мини-карта" (строки 42-44).

  Предлагаю заменить:
  Было: "Мини-карта показывается по умолчанию..."
  Стало: "Мини-карта по умолчанию скрыта..."

  Применить? (1/2/3/4)

Ты: 1

Claude:
  Обновил docs/user/Manual.md.
  Поставил в staging.
  Возвращаюсь к git-commits.

Claude (git-commits):
  Готово сообщение коммита:
  ---
  feat(settings): сделать мини-карту скрытой по умолчанию

  Изменён дефолт настройки minimap_enabled с True на False.
  Minimap теперь включается вручную через меню вида.
  Manual.md обновлён в соответствии с новым поведением.
  ---
  Коммитить?

Ты: да

Claude: (делает git commit с обоими файлами)
  Коммит 7ab3f12 создан.
```

## Related skills

- `sprint-workflow` — триггер «коммить» начинает цепочку
- `git-commits` — финальный коммит после обновления docs
- `task-tracking` — ведёт RELEASE.md (не Manual.md), работает по «дальше» и «закрываем спринт»
