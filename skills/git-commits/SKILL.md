# Git commit rules

All git commits must follow these rules strictly. No exceptions.

## Format

```
<type>(<scope, optional>): <короткое описание на русском, до 72 символов>

<развёрнутое объяснение на русском: зачем это нужно, что меняет,
какие побочные эффекты. Полными предложениями.
Одна пустая строка отделяет тему от тела.>

<опционально: BREAKING CHANGE, Refs #123, Co-authored-by>
```

## Allowed types (English only)

- `feat` — новая функциональность для пользователя
- `fix` — исправление бага
- `refactor` — рефакторинг без изменения поведения
- `perf` — оптимизация производительности
- `test` — добавление или правка тестов
- `docs` — документация (README, комментарии, docstrings)
- `style` — форматирование, пробелы, точки с запятой (без логики)
- `chore` — служебные изменения (зависимости, конфиги, CI)
- `build` — изменения в сборке, Dockerfile, Makefile
- `ci` — изменения CI/CD пайплайнов
- `revert` — откат предыдущего коммита

## Scope (optional)

Область изменений в круглых скобках после типа. Примеры scope:
- название Django app: `(measurements)`, `(accounts)`, `(fields)`
- технический слой: `(models)`, `(views)`, `(templates)`, `(api)`
- инфраструктура: `(deps)`, `(docker)`, `(ci)`, `(settings)`

Если изменение затрагивает несколько областей или общее — scope опустить.

## Rules

1. **Subject line:** до 72 символов, на русском, в инфинитиве («добавить», «исправить», «обновить»). Без точки в конце. Без заглавной буквы после двоеточия.
2. **Body:** обязателен для любого нетривиального коммита. Объясняет **зачем**, а не просто **что** (что видно из diff). Перенос строк около 72 символов.
3. **Пустая строка** между subject line и body — обязательна.
4. **Один коммит — одно логическое изменение.** Не смешивай рефакторинг с новой фичей. Не коммить миграции вместе с бизнес-логикой, если это разные задачи.
5. **Запрещены односложные сообщения:** `fix`, `update`, `wip`, `changes`, `improvements`, `улучшения`, `правки`.
6. **Нейтральный описательный стиль в инфинитиве.** Не «я сделал», «мы добавили», «исправил» — пиши «добавить», «исправить», «обновить».
7. **Footer** (опционально): после пустой строки в конце — `Refs #123`, `Closes #45`, `BREAKING CHANGE: <описание>`.

## Workflow

Когда пользователь просит сделать коммит:

1. Выполни `git status` и `git diff --staged` (если нет staged — `git diff`).
2. Проанализируй изменения:
   - какой `type` подходит (feat/fix/chore/...);
   - какой `scope` (если явно выражен);
   - что по сути меняется и **зачем**.
3. Сформулируй subject line (до 72 символов, инфинитив, по-русски).
4. Напиши body: 2-5 предложений про мотивацию, контекст и ключевые изменения.
5. **Покажи пользователю подготовленное сообщение целиком** в таком формате:

   ```
   Готово сообщение коммита:

   ---
   <subject>

   <body>
   ---

   Коммитить?
   ```

6. **Жди явного подтверждения**: «да», «ок», «коммить», «го». Молчание или уточняющий вопрос — не подтверждение.
7. Только после «да» выполни `git commit -m "<subject>" -m "<body>"`.

## Pre-commit hook failure recovery

Если `git commit` падает с ошибкой вида `files were modified by this hook`
(ruff format, black, trim whitespace и т.п.) — это **нормальное, ожидаемое
поведение**. Hook автоматически исправил форматирование, но по дизайну
pre-commit требует чтобы исправления были явно застейжены.

### Правильная процедура восстановления

1. Прочитать вывод pre-commit — понять **какие hooks** упали и **какие файлы**
   они изменили.
2. `git status --short` — убедиться что файлы действительно изменены в рабочем
   дереве после hook'а (видны как `MM` или `AM`).
3. `git add` **на те же файлы** которые были в предыдущем `git add`. Новые файлы
   добавлять НЕ нужно.
4. Повторить `git commit` с **тем же** сообщением.

Если коммит снова падает с теми же файлами — это уже реальная проблема
(например, hook не умеет автофиксить что-то), разбираемся отдельно.

### Что НЕЛЬЗЯ делать

❌ **Запускать форматтер вручную параллельно с pre-commit.** Это создаёт гонку
между staged и working tree — pre-commit стэшит unstaged, прогоняет hooks на
staged, делает stash pop, и если в working tree уже были ручные правки
форматтера — возникает `MM`/`AM` конфликт.

❌ **Лезть в `~/.cache/pre-commit/`.** Это внутренний кеш pre-commit с изолированными
установками инструментов. Запускать оттуда бинарники напрямую — обход нормального
workflow, ломает reproducibility.

❌ **Запускать `ruff format` / `black` / любой форматтер через `docker compose run`
во время процедуры коммита.** Docker контейнер работает в рабочем дереве,
pre-commit — со staged версией. Версии расходятся, возникает гонка.

❌ **Добавлять лишние файлы в `git add`.** Если pre-commit поправил файл X,
делаем `git add X`, а НЕ `git add .` — это может захватить посторонние изменения.

### Пример правильного сценария

```bash
# На MacBook, в корне проекта
$ git add src/apps/accounts/views/registration.py tests/accounts/test_x.py
$ git commit -m "feat(sp02): ..." -m "..."

# pre-commit падает:
# ruff format..........Failed
# - files were modified by this hook

$ git status --short
# MM src/apps/accounts/views/registration.py
#  M tests/accounts/test_x.py     <- ruff поправил только views/, не тесты

$ git add src/apps/accounts/views/registration.py
$ git commit -m "feat(sp02): ..." -m "..."
# ✅ hooks passed, коммит создан
```

### Когда hook падает с legitimate ошибкой (не автофикс)

Если hook пишет про ошибку которую он **не может** автофиксить (например, `ruff check` нашёл реальную проблему которую нельзя починить `--fix`):

1. НЕ пытаться обойти hook через `--no-verify`
2. Показать пользователю сам вывод hook'а
3. Дать пользователю решить: чинить проблему или осознанно пропустить hook

## Examples

### Пример 1 — новая фича

```
feat(measurements): добавить модель PenetrometerReading с PostGIS

Создана модель для хранения точечных замеров пенетрометра. Использует
PointField из PostGIS для геопривязки, depth_cm и resistance_mpa —
основные измеряемые величины. Добавлены индексы на field_id и
created_at для быстрых агрегатных запросов по участку.

Refs #12
```

### Пример 2 — баг-фикс

```
fix(auth): корректно обрабатывать истёкший JWT на refresh endpoint

До этой правки refresh endpoint возвращал 500 при передаче истёкшего
refresh-токена из-за непойманного jwt.ExpiredSignatureError. Теперь
возвращается 401 с кодом "token_expired", чтобы фронт корректно
инициировал повторный логин.
```

### Пример 3 — обновление зависимостей

```
chore(deps): обновить django с 5.0.4 до 5.0.8

Патч-апдейт с багфиксами безопасности (CVE-2024-41989, CVE-2024-41990).
Ломающих изменений нет, прогон pytest чистый.
```

### Пример 4 — рефакторинг

```
refactor(models): вынести валидацию координат в отдельный миксин

Валидация широты/долготы дублировалась в трёх моделях (Field,
MeasurementPoint, Tractor). Вынесена в GeoValidationMixin для
единообразия и упрощения тестов. Поведение не меняется.
```

### Пример 5 — инициализация проекта

```
chore(setup): инициализировать структуру проекта и Claude Code setup

Создан скелет Django-проекта с базовой структурой директорий (src,
tests, docs). Добавлена конфигурация Claude Code: проектный
.claude/settings.json с Django-specific permissions, CLAUDE.md с
описанием стека и task workflow, .claudeignore для защиты контекста
от шума. Миграции коммитятся, static/media — игнорируются.
```

### Пример 6 — breaking change

```
feat(api)!: переименовать поле depth_cm в depth_mm

BREAKING CHANGE: API теперь принимает и возвращает глубину в
миллиметрах вместо сантиметров для консистентности с данными
от датчика пенетрометра. Клиенты должны умножить все значения
на 10 при обновлении.

Refs #47
```

## Запрет на Co-Authored-By

НЕ добавлять в commit message строки:
- Co-Authored-By: Claude ...
- 🤖 Generated with Claude Code
- Любые автопометки о том что коммит сделан с помощью AI-ассистента

Причина: Vladimir — единственный автор коммитов в своих проектах. Атрибуция через Co-Authored-By и подобные строки создаёт ложное впечатление многоавторства и загрязняет историю git. Инструменты автоматизации (включая Claude Code) не являются соавторами в семантическом смысле.

Если в шаблонах или дефолтах Claude Code такие строки появляются автоматически — удалять перед git commit -m.

Применяется ко ВСЕМ проектам, без исключений.

После правки показать diff skill-файла. НЕ коммитить (это файл в ~/.claude/, вне git репо проектов).

## Anti-patterns — НЕ делай так

❌ `fix: баг` — нет описания, что за баг
❌ `update README` — какой update? зачем?
❌ `wip` — не коммитим work-in-progress в main/dev ветку
❌ `changes` — бессмысленно
❌ `feat: added new feature` — описание должно быть на русском
❌ `feat: Добавил новую модель.` — с заглавной, с точкой, от первого лица
❌ `feat(models): добавить модель Measurement и исправить баг в сериализаторе` — два логических изменения в одном коммите
❌ `feat: добавить очень длинное описание которое превышает семьдесят два символа и уезжает за край` — subject line слишком длинный
❌ Отсутствие body у нетривиального коммита

## Special cases

### Если пользователь просит «быстрый коммит»
Всё равно применяй формат. «Быстрый» ≠ «плохой». Сократи body до 1-2 предложений, но не опускай полностью, если изменение нетривиальное.

### Если изменений много и они разнородные
Предложи пользователю разбить на несколько коммитов. Покажи логическую группировку файлов:

```
Я вижу изменения в нескольких областях. Предлагаю разбить на 3 коммита:

1. feat(measurements): добавить модель PenetrometerReading
   — src/measurements/models.py, src/measurements/migrations/0002_*.py

2. test(measurements): тесты для PenetrometerReading
   — tests/measurements/test_models.py, tests/factories/measurements.py

3. docs: документация по структуре замеров
   — docs/data-model.md

Разбивать или коммитить одним?
```

### Если есть staged и unstaged одновременно
Коммить только staged, unstaged не трогай. Явно скажи пользователю:
«Вижу также изменения в `<files>`, которые не в staging. Оставляю их в рабочей копии.»

### Если ветка защищённая (main/master/production)
Если определил по `git branch --show-current`, что ветка — `main`, `master` или `production` — **предупреди** пользователя перед коммитом: «Коммит напрямую в защищённую ветку `<branch>`. Обычно работа идёт через feature-ветки и PR. Продолжить?»

## Constraints

- **Никогда не выполняй `git push`** — даже после успешного коммита. Push делает пользователь вручную.
- **Никогда не делай `git commit --amend`** без явного запроса пользователя — это перезапись истории.
- **Никогда не коммить автоматически** без показа сообщения и подтверждения.
- Если не уверен в `type` или `scope` — спроси пользователя, не угадывай.
