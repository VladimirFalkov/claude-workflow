# Sprint workflow — основной цикл разработки

Главный skill — как мы работаем. Применяется во всех проектах.

## Принципы

1. Agile-спринты. Один спринт = один логический блок.
2. **TDD строго** — тест first, потом код. Исключения: migrations, settings, template markup.
3. Приоритет: скорость к деплою + best practices. Критичное чиним сейчас, остальное — в backlog.
4. Не писать код пока не получены все нужные файлы.
5. **Claude никогда не коммитит автоматически.** Только по команде «коммить».
6. **Claude никогда не делает `git push`.**
7. Экономим токены. Крупные шаги — в `implementer` subagent.
8. **Не спрашиваем подтверждение на создание файлов которые есть в плане** (DONE.md, RELEASE.md, BACKLOG.md, директории docs/user, docs/dev и т.п.). Эти пути в allow permissions.

## Структура проекта

```
.project-state/
├── roadmap.md
├── backlog-global.md
└── sprints/spNN_<slug>/
    ├── 00-plan.md           ← исходный план от Владимира
    ├── 01-spec.md           ← от spec-pm (опционально)
    ├── 02-architecture.md   ← от architect
    ├── 03-plan.md           ← от planner
    ├── 05-review.md         ← от reviewer (AUTO после каждого шага)
    ├── DONE.md              ← task-tracking по «дальше»
    ├── BACKLOG.md           ← task-tracking по «в backlog»
    └── RELEASE.md           ← task-tracking по «дальше»

docs/
├── user/Manual.md, changelog.md
├── dev/architecture.md, data-model.md, patterns.md
├── adr/NNNN-title.md
└── reference/...
```

## Основной цикл задачи

```
1. Запрос файлов
2. TDD-код (тест + минимальная реализация + refactor)
3. ► reviewer subagent AUTO ◄    ← сразу после выдачи кода
4. Claude фиксит BLOCKER и MAJOR из reviewer report
5. Твой ручной аудит 🔴🟡🟢 поверх
6. Claude применяет фиксы
7. «дальше» → task-tracking (DONE, RELEASE). Claude ПРЕДЛАГАЕТ коммит:
   ├─ не последний шаг → step-коммит (feat(spNN): Шаг N — <что>)
   └─ последний шаг → step-коммит + артефакты закрытия одним коммитом
8. Владимир отвечает (да / нет / позже). Если «да» → «коммить» выполняется.
9. Следующая задача / следующий спринт
```

## Шаг 1 — Запрос файлов

- Проси **все нужные файлы сразу**, не по одному.
- Если паттерн виден из уже прочитанных — не проси повторно.
- Уточняющие вопросы — ДО кода, один раунд максимум.

**Anti-pattern:** начать писать, обнаружить нехватку, запросить, залепить заплаткой. Если понял в процессе — остановись и **перепиши заново**.

## Шаг 2 — TDD-порядок (строго)

**Правила написания тестов:** skill `testing-rules` (global anti-patterns) + `testing-django` (stack-specific). Применяются в каждом RED-GREEN-REFACTOR цикле. Без них тесты часто получаются тавтологичными или слабыми.

**Red → Green → Refactor.**

1. **Red.** Написать тест. Запустить. Убедиться что падает **по ожидаемой причине**.
2. **Green.** Минимальная реализация. Тест проходит.
3. **Full suite.** Весь pytest — должен быть **полностью зелёным**, не только «мои тесты зелёные».

   **Правило по красной сьюте (broken-before):**

   Если после реализации шага в full suite есть упавшие тесты — Claude НЕ пишет `READY_FOR_MERGE` и НЕ идёт к следующему шагу. Диагностирует и действует по алгоритму:

   **A. Упавшие тесты связаны с текущим шагом** (те же модули / та же функциональность) — фиксит в рамках шага, как часть обычного TDD-цикла.

   **B. Упавшие тесты НЕ связаны с текущим шагом** («broken-before», упали ещё до начала Шага N):

   Три варианта — выбирает Claude сам по эвристике ниже, сообщает Владимиру что выбрал и почему:

   1. **Починить в текущем шаге** — если фикс локальный (1-2 файла, ≤10 строк, причина ясна из traceback). Пример: сломался импорт, заменился путь settings.
   2. **Sub-step перед продолжением** — если починка нетривиальная (меняет логику, добавляет тесты, затрагивает 3+ файла). Создаёт `Шаг N.fix: починить <что>` со своим TDD-циклом и отдельным step-коммитом. Только ПОСЛЕ него — Шаг N+1.
   3. **Skip с записью в BACKLOG** — если починка большая И не блокирует разработку текущего спринта. Ставит `@pytest.mark.skip(reason="broken since spXX, see BACKLOG-NNN")`, **немедленно** создаёт запись в BACKLOG.md с приоритетом 🔴 если тест покрывал security/корректность, 🟡 если feature, 🟢 если косметика. **Skip без записи в BACKLOG запрещён.**

   **Запрещено строго:**
   - Писать `READY_FOR_MERGE` при красной сьюте
   - Фраза «не моё, не трогал» как финальный ответ — это эскалация Владимиру, а не закрытие вопроса
   - `--deselect` / `-k "not test_broken"` в команде запуска — прятание проблемы
   - Добавлять `@pytest.mark.skip` без записи в BACKLOG в том же коммите

   **Сообщение Владимиру** в формате summary шага — добавляется блок:
4. **Refactor.** Тесты остаются зелёными.

Один тест — один концепт. Не пихать 5 assert'ов.

### Когда TDD НЕ применяется (но проверка всё равно нужна)

| Изменение | Проверка |
|---|---|
| Django migration | `makemigrations --dry-run`, `migrate --plan` |
| Settings | старт shell/runserver, явная проверка значения |
| Template markup без логики | ручная UX-проверка Владимиром |
| Чистый рефакторинг | полный pytest — поведение идентично |
| Data migration | отдельный тест + dry-run на тестовой БД |

Сомнение → TDD применяется.

### Формат выдачи кода

**Один файл — один блок кода.** Никогда два файла в одном блоке.

**Новый файл** — полный текст, первая строка — путь комментарием.

**Редактирование** — Было/Стало с контекстом (имя функции).

**Фикс по аудиту** — первой строкой `Файл: ...`.

**Много разбросанных правок** — полный текст файла.

## Scope дисциплина внутри шага

Каждый шаг из `03-plan.md` — **атомарен**. Код одного шага касается только файлов и логики, явно указанных в плане для этого шага.

### Правило

Если при реализации Шага N Claude обнаруживает что требуется код из Шага N+k:

1. **СТОП.** Не писать "заодно".
2. Сообщить Владимиру: «Шаг N требует X из Шага N+k, потому что ...»
3. Владимир решает:
   - **Склеить шаги** — реализовать N+N+k как один TDD-цикл
   - **Pass-through skeleton** — оставить заглушку для N+k, которая не ломает код
   - **Переплан** — пересмотреть 03-plan.md

### Anti-patterns

- **Stub-porn** — «stub модель User / stub middleware с логикой на 50%». Либо полная реализация, либо пустой pass-through. Не половина.
- **Миграция 0002 для фикса 0001 до коммита** — удалить 0001 через `git rm` и перегенерировать. Чистая история миграций на шаг.
- **"Раз уже лезем в соседний файл — заодно..."** — даже мелкое расширение scope создаёт нечитаемый diff для reviewer'а и блокирует откат.

### Как это проверяется

- **Reviewer** смотрит diff: если видит код не из текущего шага — BLOCKER.
- **Claude main loop** — при запуске шага явно формулирует: «Шаг N атомарен: только <список>, без <список что НЕ делаем>».

### Пример

**Плохо:**
```
Шаг 1 по плану: bootstrap accounts app, skeleton middleware.
Claude: создаёт полную User модель + AUTH_USER_MODEL в settings + backends.py.
Результат: залез в Шаги 2 и 5, pytest падает, миграция нужна но её нет,
откат через несколько попыток. Потеря времени и $.
```

**Хорошо:**
```
Шаг 1 по плану: bootstrap accounts app, skeleton middleware.
Claude: создаёт пустой apps/accounts/, пустой middleware (pass-through).
Остановился. Шаг 2 (модели + миграция) — следующий отдельный TDD-цикл.
```

## Шаг 3 — AUTO reviewer после кода

**После того как шаг TDD завершён** (тест написан, код написан, full suite зелёный) — **Claude автоматически вызывает `reviewer` subagent**. Не ждёт команды от Владимира.

### Что делает reviewer

- Читает `git diff` текущих незакоммиченных изменений (или рабочего дерева, если до коммита)
- Пишет `.project-state/sprints/spNN_<slug>/05-review.md` через Write tool с замечаниями по severity: BLOCKER / MAJOR / MINOR / NIT
- Возвращает сводку в main-контекст в стандартном формате

### Что делает Claude после reviewer

1. Если есть **BLOCKER** — молча применяет фиксы и **перезапускает reviewer** (до 2 итераций max, потом эскалация Владимиру).
2. Если есть **MAJOR** — применяет фиксы молча.
3. Если есть **NIT** — применяет фиксы молча. В summary помечает пофикшенные ✅.
4. **MINOR** — решение принимает Claude сам по правилам ниже:

   **Фиксит молча (без вопроса Владимиру), если выполнено ЛЮБОЕ:**
   - **«Здесь»** — MINOR в файле/функции/модуле, которую Claude прямо сейчас редактирует в рамках текущего шага.
   - **«Архитектурный»** — MINOR про валидаторы, constraints, контракты между модулями, миграции, API — то есть то, что пробьётся в архитектуру или создаст техдолг, дорогой в починке позже.
   - **«Ответ уже есть в 02-architecture.md»** — решение по фиксу зафиксировано в архитектурном документе спринта, угадывать ничего не нужно.

   **Откладывает в BACKLOG (молча, без вопроса), если выполнены ВСЕ:**
   - Не в файле/функции текущего шага.
   - Косметический (docstring, имя переменной, локальный рефакторинг внутри метода, форматирование).
   - В roadmap **нет** спринта где мы снова будем работать с этим кодом в ближайшие 2-3 спринта.

   **Эскалирует Владимиру вопросом ТОЛЬКО если:**
   - Фикс требует архитектурного решения, которого **нет** в `02-architecture.md` (например, формат поля не зафиксирован).
   - Фикс меняет scope шага — тянет правки в другие модули или следующие шаги плана.
   - Есть два разумных варианта с разными trade-offs, и выбор не очевиден из контекста.

   В summary Владимиру помечает: ✅ пофикшенные, 📝 ушедшие в BACKLOG, ❓ эскалированные вопросом.


### Что Claude показывает Владимиру

Reviewer возвращает стандартный summary — Claude передаёт его Владимиру **без сокращений**:

```
Шаг N реализован.

TDD cycle: ✅ (test → red → green → full suite pass)

Review written: .project-state/sprints/spNN_<slug>/05-review.md

Status: READY_FOR_MERGE | NEEDS_CHANGES
Counts: BLOCKER=X, MAJOR=X, MINOR=X, NIT=X

BLOCKER (X): [список если есть; пофикшенные помечены ✅]
MAJOR (X): [список если есть; пофикшенные помечены ✅]
MINOR (X):
1. <title> (file.py:42) — <fix hint>
2. ...
NIT (X): [пофикшенные помечены ✅]
1. <title> (file.py:42) — <fix hint>
2. ...

Жду ручной аудит.
```

### Когда пропустить AUTO reviewer

- Шаг тривиальный (правка опечатки, docstring, одна строка)
- Шаг — применение фикса по предыдущему аудиту (не новый код)
- Явная просьба Владимира «без ревью»

В этих случаях Claude пишет: «Пропускаю AUTO reviewer: <причина>».

## Шаг 4 — Ручной аудит Владимира

Подробности — в skill `audit-cycle`. Работает поверх того что уже пофиксил reviewer.

Коротко:
- Владимир присылает 🔴🟡🟢 замечания
- Claude **во время аудита код не пишет** — ждёт полного списка
- После «применяй» — targeted fixes

## Шаг 5 — Баги по ходу

Если Claude находит проблему в чужом коде — **не молчит**:
1. Сообщает: что, где
2. Владимир решает: чиним / backlog
3. Дефолт — чиним сейчас если ≤2 файла

## Когда делегировать в `implementer` subagent

### Делегируем когда:
- Шаг `03-plan.md` затрагивает **4+ файлов**
- Нужен **exploratory поиск** по codebase
- Главный контекст **60%+ заполнен**
- Нужно читать **много крупных файлов**

### НЕ делегируем когда:
- Правка в 1-3 файлах, файлы уже в контексте
- **Фикс по аудиту** (🔴🟡) — всегда main loop
- Опечатка, комментарий, docstring
- Добавление теста к готовому коду

### Как

```
Реализуй Шаг N из .project-state/sprints/spNN_<slug>/03-plan.md.
Output: сводка изменённых файлов + статус тестов.
НЕ коммить. НЕ писать в docs. НЕ идти в следующий шаг.
Scope дисциплина — только файлы Шага N, без захода в следующие.
```

### После возврата implementer'а

1. Main loop читает `git diff` (не файлы целиком)
2. **AUTO reviewer** на diff — как обычно
3. Показывает Владимиру результат + reviewer report
4. Ручной аудит и фиксы — делает main loop

## Триггерные команды Владимира

### «дальше» / «go» / «следующая задача»

Задача принята. Claude:

1. Активирует `task-tracking`:
   - Дописывает `DONE.md`
   - Если было user-facing — дописывает `RELEASE.md`
2. **НЕ перезапускает reviewer** (он уже был после кода).
3. **НЕ коммитит автоматически.**
4. **Проверяет позицию в плане:**
   - Посчитать шагов в `03-plan.md` (X)
   - Посчитать закрытых в `DONE.md` (Y)
5. **Если Y < X (не последний шаг):** предлагает step-коммит сейчас:

```
   ✅ Шаг N закрыт (DONE.md обновлён, Y из X).

   Сделать step-коммит сейчас?
     1. да — step-коммит (feat(spNN): Шаг N — <что>)
     2. нет — не коммитить, сразу к следующему шагу
     3. позже — продолжить работу, коммит сделаю потом сам

   Выбор?
```

   **По умолчанию рекомендация — вариант 1** (step-коммит сейчас, без накопления).

6. **Если Y == X (последний шаг спринта):** предупреждает и предлагает closure-коммит:

```
   ✅ Шаг X закрыт. Все X шагов spNN выполнены — план завершён.

   Это последний шаг. Предлагаю closure-коммит:
   код Шага X + артефакты закрытия (changelog, Manual, ADR) одним коммитом.

     1. да — сгенерировать артефакты закрытия и сделать один коммит (рекомендуется)
     2. только код Шага X — step-коммит сейчас, закрытие спринта позже
     3. позже — ничего не коммитить

   Выбор?
```

7. **Ждёт ответ.** Только после ответа:
   - Вариант «да» (пункт 5) → step-коммит по Случаю А (см. «коммить»)
   - Вариант «да» (пункт 6) → closure по Случаю Б (см. «коммить»)
   - Варианты «нет»/«позже» → предлагает следующий шаг из плана или выходит

### «коммить» / «commit» — С РАЗВИЛКОЙ

Коммит может инициироваться двумя путями:

**Путь 1 — автоматически через «дальше».** После закрытия шага (Y < X) или
закрытия последнего шага (Y == X) команда «дальше» САМА предлагает коммит
(см. секцию «дальше»). Это предпочитаемый путь — не забываешь коммитить.

**Путь 2 — явно через «коммить».** Пользователь прямо набирает «коммить».
Claude проверяет статус плана и выбирает Случай А или Случай Б
(логика ниже такая же, как в «дальше»).

#### Шаг 1 — Проверка статуса спринта

1. Определить активный спринт: `.project-state/sprints/spNN_<slug>/`
2. Прочитать `03-plan.md` — посчитать шагов в плане (X)
3. Прочитать `DONE.md` — посчитать закрытых шагов (Y)
4. Сравнить X и Y

#### Шаг 2 — Развилка

**Случай А — не последний шаг спринта (Y < X): step-коммит**

Простой коммит по текущему шагу. Код шага + его тесты + служебные файлы
(`.project-state/sprints/spNN_<slug>/DONE.md`, `BACKLOG.md`, `05-review.md`,
`RELEASE.md`). **Включает `.claude/settings.json` если правился.**
Никаких артефактов закрытия спринта.

1. `git status`, `git diff --stat` — проверка что в staging файлы текущего шага
2. **НЕ активирует `docs-sync`** (он для закрытия спринта)
3. Активирует `git-commits` в режиме **step-commit**. Формат сообщения:

```
   feat(spNN): Шаг N — <краткое описание>

   <1-3 строки что сделано: модели / сервисы / тесты>

   Refs: .project-state/sprints/spNN_<slug>/
```

   Примеры:
   - `feat(sp02): Шаг 10 — Django формы auth`
   - `feat(sp02): Шаг 11 — URL routing`
   - `feat(sp02): Шаг 12 — LoginView с rate limiting`

4. Показывает сообщение, ждёт «да»
5. **Один `git commit`** — код текущего шага
6. **НЕ push.**

**Случай Б — последний шаг спринта (Y == X): step-коммит + артефакты закрытия одним коммитом**

Код последнего шага **вместе** с closure-артефактами (changelog, Manual, ADR, обработанный BACKLOG). Один коммит.

Спросить Владимира:

```
Все шаги spNN закрыты (X из X). План выполнен.

Коммитим последний шаг + артефакты закрытия спринта одним коммитом?
  1. да — один коммит (код Шага X + changelog + Manual + ADR)
  2. только код — step-коммит сейчас, закрытие отдельно позже
  3. позже — ничего не коммитить

Выбор?
```

**Ждёт ответ.** Никаких действий до ответа.

#### Шаг 3 — Обработка ответа (Случай Б)

**Если «да» (один финальный коммит со всем):**

1. Активирует `task-tracking` «закрываем спринт»:
   - Мёржит `RELEASE.md` → `docs/user/changelog.md`
   - Обновляет `Manual.md` (спрашивает по пунктам если есть затронутое)
   - Разбирается с `BACKLOG.md` (один вопрос: куда перенести)
   - Предлагает `ADR` (один вопрос со списком)
2. Активирует `docs-sync` — финальная проверка docs
3. Активирует `git-commits` — commit message охватывает **код последнего шага и артефакты закрытия**:

```
   feat(spNN): Шаг X — <название шага> + закрытие спринта NN

   Завершён спринт NN — <название спринта>. Закрыто X задач.

   Шаг X:
   - <что реализовано в финальном шаге>

   Артефакты закрытия спринта:
   - docs/user/changelog.md — раздел Sprint NN
   - docs/user/Manual.md — обновлено <X секций> (если было)
   - docs/adr/NNNN-*.md — добавлены ADR <номера> (если создавали)

   Refs: .project-state/sprints/spNN_<slug>/
```

4. Показывает сообщение, ждёт «да»
5. **ОДИН `git commit`** — код последнего шага + артефакты закрытия
6. **НЕ push.**

**Если «только код»:**

Step-коммит по Случаю А (только код последнего шага). Артефакты закрытия сделаешь отдельно позже командой «закрываем спринт».

**Если «позже»:**

Ничего не делает. Выходит без коммита.

### «в backlog» / «отложить»

`task-tracking` пишет в BACKLOG.md (спринта или глобальный).

### «закрываем спринт» / «финализируем спринт N»

Отдельная команда для случая когда Владимир хочет закрыть спринт **без коммита сейчас** (например, чтобы проверить артефакты перед коммитом). `task-tracking` отрабатывает процедуру одним проходом без подтверждений на каждый шаг.

После этого Владимир делает «коммить» — создаст closure-коммит с артефактами.

## Когда использовать агентов

| Задача | Агент | Когда |
|---|---|---|
| Идея → ТЗ | `spec-pm` | Нет спеки |
| Спека → архитектура | `architect` | Нужен ADR |
| Архитектура → шаги | `planner` | Нужны атомарные задачи |
| Крупный шаг | `implementer` | 4+ файлов / exploratory / забит контекст |
| Поиск по коду | `researcher` | «как работает X», «где определён Y» |
| Тесты | `test-runner` | Ручной прогон, «запусти тесты» |
| Ревью | `reviewer` | **AUTO после кода**, ручной запуск по запросу |

### НЕ агенты (main loop):
- Правка 1-3 файлов → main loop
- Фикс опечатки / docstring → main loop
- Фикс по 🔴🟡 замечаниям → main loop
- Ответ на вопрос с контекстом → main loop

## Что НЕ делает Claude

- **Не пушит** (git push запрещён)
- **Не делает `git commit --amend`** без запроса
- **Не коммитит** автоматически — только после подтверждения Владимира
  («да» на предложение step-коммита из «дальше», или явное «коммить»)
- **Не оставляет несколько шагов в одном коммите** — каждый шаг = свой step-коммит.
  Если Владимир пропустил коммит и накопилось 2+ шага в working tree — при
  следующем «коммить» Claude ПРЕДУПРЕЖДАЕТ и предлагает разбить на отдельные
  коммиты по файлам шагов (по `DONE.md` определить какие файлы к какому шагу)
- **Не пишет код** пока не прочитаны все файлы
- **Не пишет код** во время аудита — ждёт
- **Не пишет код** без сначала падающего теста (исключения выше)
- **Не молчит** про найденные баги
- **Не делает** rm -rf, sudo, install, migrate (в deny)
- **Не использует** `python3 -c` для операций с файлами (os.remove, shutil.rmtree и т.п.) — это обход security настроек. Использует `git rm`, `rm`, `git clean -f`.
- **Не спрашивает** подтверждение на создание DONE/BACKLOG/RELEASE/changelog — создаёт молча
- **Не активирует docs-sync на промежуточных step-коммитах** — только при закрытии спринта
- **Не задаёт вопросов по MINOR если ответ очевиден** — фиксит сам если MINOR в текущем файле/функции шага ИЛИ архитектурный ИЛИ решение есть в 02-architecture.md. Откладывает в BACKLOG молча если косметический и не в текущем scope. Эскалирует только при реально неоднозначном выборе.
- **Не лезет в будущие шаги** — scope дисциплина (см. секцию выше)
