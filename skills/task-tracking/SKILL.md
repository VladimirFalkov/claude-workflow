# Task tracking — ведение DONE/BACKLOG/RELEASE

Управляет файлами состояния спринта в `.project-state/sprints/spNN_<slug>/`. Append-only история: что сделано, отложено, релизнуто.

## Триггеры

| Фраза | Что делает |
|---|---|
| «дальше», «go», «следующая задача» | Дописывает DONE.md, опционально RELEASE.md, предлагает next task |
| «в backlog», «отложить», «потом» | Дописывает BACKLOG.md |
| «закрываем спринт», «финализируем спринт N» | Мёржит RELEASE → changelog, обновляет Manual, предлагает ADR |

## Где ищем активный спринт

1. `.project-state/sprints/` — берём папку с самым большим номером `spNN_<slug>/`
2. Если не определён — спросить: «Не могу определить активный спринт. Какой?»
3. Файлы DONE/BACKLOG/RELEASE создаются автоматически при первой записи, **без подтверждения**

---

## Триггер «дальше»

### Шаг 1 — Состояние

- `git status`, `git log -1 --stat`
- Определить что сделано с последнего «дальше»

### Шаг 2 — Reviewer уже был (AUTO после кода)

Reviewer запускается **не здесь**. Он работает автоматом после каждого шага TDD (см. `sprint-workflow`). По «дальше» его НЕ перезапускаем.

Но проверяем: существует ли `05-review.md` для текущего шага. Если нет — reviewer пропустили, запускаем его ОДИН раз сейчас.

### Шаг 3 — Дописать DONE.md

Файл: `.project-state/sprints/spNN_<slug>/DONE.md` — создаётся без подтверждения если нет.

Первый раз в спринте — заголовок:

```markdown
# Sprint NN — <название>: Готово

<YYYY-MM-DD>
```

Затем блок в конец:

```markdown
## <короткое название задачи>
**Дата:** <YYYY-MM-DD>
**Commits:** `<hash1>`, `<hash2>` (если уже есть)
**Файлы:** `path/a.py`, `path/b.py`, `tests/test_x.py`

<1-3 предложения: что сделано, зачем, ключевые решения>

**Проверено:** <pytest path, ручная проверка>
```

### Шаг 4 — Если user-facing — дописать RELEASE.md

**Пишем в RELEASE:**
- Новая/изменённая UI-фича
- Изменилось поведение видимое пользователю
- Новый endpoint публичного API
- Изменилось поведение по умолчанию

**Пропускаем:**
- Рефакторинг
- Новые тесты без нового поведения
- Служебные migrations
- Чисто внутренние изменения

Файл создаётся без подтверждения. Формат:

```markdown
# Sprint NN — <название>: Release notes (draft)

Финальная версия уходит в docs/user/changelog.md при закрытии.
```

Блок в конец:

```markdown
## <Название фичи для пользователя>

<1-2 предложения простым языком, без технических деталей>
```

**Язык RELEASE.md — пользовательский.** «Добавлена возможность скрывать мини-карту» ✅. «Реализован toggle через context processor» ❌.

### Шаг 5 — Сводка + проверка завершённости спринта

Смотрим на `.project-state/sprints/spNN_<slug>/03-plan.md`:
- Сколько всего шагов
- Сколько из них есть в DONE.md

Формат ответа Владимиру:

```
✅ Задача «<название>» закрыта.

DONE.md: +1 запись  (всего X из Y шагов плана)
RELEASE.md: +1 пользовательское  (или: без user-facing)

<Если X < Y:>
Следующая задача по плану 03-plan.md:
— **Шаг N+1:** <название>
Начинаем?

<Если X == Y:>
⚡ Все шаги sp01 закрыты (X из X). План выполнен.
Можно сразу «коммить» — после коммита предложу закрыть спринт.
```

**НЕ коммитит.** Ждёт «коммить».

## Триггер «в backlog»

### Scope

Спросить (или понять):
- Текущий спринт или глобально?
- Непонятно → дефолт текущий спринт

### Запись

**Текущий спринт:** `.project-state/sprints/spNN_<slug>/BACKLOG.md`
**Глобально:** `.project-state/backlog-global.md`

Файл создаётся без подтверждения.

Заголовок первый раз:

```markdown
# Sprint NN — <название>: Backlog

Задачи, отложенные на потом внутри этого спринта.
```

Блок:

```markdown
## <краткое название>
**Дата:** <YYYY-MM-DD>
**Источник:** <из ревью / из аудита / Владимир предложил / найдено в коде>
**Приоритет:** 🔴 critical / 🟡 important / 🟢 nice-to-have

<1-3 предложения>

**Примерный scope:** <файлы, оценка small/medium/large>
```

### Подтверждение

Коротко:

```
📝 В backlog: «<название>» → spNN_<slug>/BACKLOG.md
```

## Триггер «закрываем спринт»

**Новый принцип:** сразу делаем всё что можно без подтверждений. Спрашиваем ТОЛЬКО по критичным точкам (Manual.md и ADR).

### Шаг 1 — Сводка (без вопроса «продолжаем?»)

```
Закрываем Sprint NN — <название>.

DONE.md:    X задач
RELEASE.md: Y user-facing изменений
BACKLOG.md: Z отложенных

Делаю сейчас:
- Мёрж RELEASE.md → docs/user/changelog.md
- Проверка Manual.md (спрошу по каждому пункту)
- Перенос BACKLOG (спрошу куда)
- Предложу ADR если есть
```

### Шаг 2 — Мёрж RELEASE → changelog (AUTO)

Файл `docs/user/changelog.md` создаётся без подтверждения если нет.

Если нет — создать с шапкой:

```markdown
# Changelog

Пользовательская история изменений в продукте.

Формат: [Keep a Changelog](https://keepachangelog.com/).
```

Добавить в **начало** (после шапки):

```markdown
## Sprint NN — <название> · YYYY-MM-DD

<содержимое RELEASE.md, возможно лёгкая доработка формулировок>
```

**Если RELEASE пустой** (инфраструктурный спринт) — добавить раздел с пометкой:

```markdown
## Sprint NN — <название> · YYYY-MM-DD

*Инфраструктурный спринт — без пользовательских изменений.*

См. `.project-state/sprints/spNN_<slug>/DONE.md` для deveплоперских деталей.
```

**RELEASE.md НЕ удалять** — остаётся как исторический артефакт.

### Шаг 3 — Manual.md (СПРАШИВАЕМ по каждому)

Проверить существует ли `docs/user/Manual.md`.

**Если нет** — спросить один раз:

```
docs/user/Manual.md не существует.

Создать? (да / нет / позже)
```

**Если есть** — Grep по изменившейся функциональности. Для каждого найденного места:

```
1. RELEASE блок: «<текст>»
   → Manual.md:42: «<текущая формулировка>»
   Предлагаю заменить на: «<новая формулировка>»
   (1 применить / 2 свой текст / 3 пропустить)
```

По одному пункту, не скопом.

### Шаг 4 — BACKLOG (один вопрос на весь спринт)

Если `BACKLOG.md` не пустой:

```
В спринтовом BACKLOG.md осталось Z задач.

Куда их переместить?
1. В .project-state/backlog-global.md (глобально)
2. В следующий спринт spNN+1 (создам skeleton)
3. Оставить в этом спринте (ничего не делать)
4. Разобрать по одной (покажу каждую)
```

**Один вопрос, один ответ.** Не 5 отдельных.

### Шаг 5 — ADR (СПРАШИВАЕМ)

Проверить:
1. Есть ли `02-architecture.md` в спринте — если да, извлечь "Chosen approach"
2. Посмотреть в `docs/architecture.md` секцию "ADR — планируемые" — какие ADR помечены на этот спринт
3. Посмотреть git log спринта на явные архитектурные решения

Показать:

```
Возможные ADR для фиксации в Sprint NN:

1. ADR-0001: Django monolith over microservices
   Источник: docs/architecture.md (планируемые ADR, sp01)
   Статус решения: принято, код отражает

2. ADR-0002: src/apps/<app> modular layout
   Источник: git log sp01, config/settings/
   Статус: принято в sp01

Какие создавать? (укажи номера через запятую, или «все», или «пропустить»)
```

После ответа — создать выбранные. Формат:

```markdown
# ADR-NNNN: <название>

- **Status:** Accepted
- **Date:** YYYY-MM-DD
- **Sprint:** NN

## Context
<почему потребовалось>

## Decision
<что решили>

## Consequences
<плюсы/минусы>

## References
- Sprint NN: .project-state/sprints/spNN_<slug>/
- Commits: <если есть>
```

### Шаг 6 — Финальная сводка

```
✅ Спринт NN закрыт.

Сделано:
- changelog.md: добавлен раздел Sprint NN
- Manual.md: <X секций обновлено / создан / пропущен>
- BACKLOG: <куда перенесён>
- ADR создано: <список номеров или "нет">

Изменённые файлы в working copy:
  docs/user/changelog.md
  docs/user/Manual.md (если правили)
  docs/adr/NNNN-*.md (если создавали)
  .project-state/backlog-global.md (если переносили туда)

Скажи «коммить» — зафиксирую финальный коммит закрытия спринта.
```

## Формат дат

`YYYY-MM-DD`. Текущая — системная, не придумывать. Если неясно — спросить.

## Append-only принцип

**Не редактировать уже записанные блоки.** Если нужна правка — новый блок:

```markdown
## [UPDATE YYYY-MM-DD] Поправка к «<название>» от YYYY-MM-DD

<что поправилось>
```

Исключение: свежая ошибка в только что записанном (опечатка) — поправить молча.

## Anti-patterns

❌ Редактировать старые записи в DONE/RELEASE
❌ Писать в changelog.md без «закрываем спринт»
❌ Пушить в RELEASE внутренние изменения
❌ Дублировать в DONE и RELEASE одинаковым текстом
❌ Коммитить автоматически после «дальше»
❌ **Спрашивать подтверждение на создание DONE/RELEASE/BACKLOG** (создаём молча)
❌ **Разбивать Manual/BACKLOG/ADR на много отдельных вопросов** (один вопрос на блок)
